I’ve had a few requests for the method that I use for encrypting posts for displaying on github so I’ve decided to make a post about it. It won’t be perfect; but nothing ever is.

I originally made this for hackthebox writeups but it could apply to anything really. So there are some slight differences between the site currently and the pictures included in the writeup since they used my hackthebox username.

The tool I will use is called staticrypt.

It takes html and encrypts it using aes-256 but you’re better using a long password since the page could be downloaded to be cracked if you hold sensitive data behind the password (eg credit card data).

Some things first:

This method could perhaps apply to similar frameworks (eg hugo) as long as you can see both the markdown (source) and the html (built) files. You can skip most of the post if you just want a passworded page but I’m also going to cover styling alongside with a basic idea of how to automate everything.

I also used this method before the cli version of staticrypt:

https://www.npmjs.com/package/staticrypt

But I will slightly modify my explanation to use the cli just in an inefficient way but this will become clearer when I cover this step.

Also, this guide will cover how I do it so will assume that you have a public github repo for your github pages. This method could be further refined if you have a private repo since you can put the unencrypted files on github and could do it all from a github action and bypass lots of this. You could also incorporate this into a github action so that you only have to upload your markdown file and github can do the rest but I am covering this with the idea that your repo is public so cannot upload unencrypted files as they would be accessible even after deletion

I don’t use this method but

xct/jekyll-encrypt

also implements a similar idea but uses a jekyll plugin so should do it automatically - however, I could not get this working.

Also, encrypting sensitive data using this method isn’t advised since you should probably use something more secure (and probably not post online in the first place).

Finally, this is by no means the most efficient or best way to encrypt posts, this is just the way I found that worked for me. There may be multiple instances in this post that are far from efficient or poor coding but this is just what worked for me and you can feel free to message me with improvements if you want. I do not expect that this method is the best way to do it but after lots of effort, this is the way that worked for me.

Maybe what I’m trying to say is this should be viewed more as “what this guy did” and get a general idea as opposed to “here’s the steps to replicate” .

Step 3 - Templating

So this is the part where I didn’t follow the expectations of the templating built into the cli. Instead of using the flag -f and specifying a template, I wrote a python script to get the encrypted string from the website and insert it into my own template.

Now that there is a cli that supports a template, you could use this however I have tweaked my method to incorporate the cli without using the templating. I also wrote my template to incorporate crypto-js and the decryption script rather than the cli embedding it for the same reason.

I complete the templating as follows:

  • Embedding images into the markdown file
  • Doing a jekyll build
  • Take the generated html and encrypting
  • Taking the encrypted string and inserting into the template.
  • Taking other needed parts (eg a section of the password as a hint of what to enter) and inserting into the template
  • Replacing the generated html file with the filled template file

As previously mentioned, I am writing this with the assumption that these files will be hosted on a public repo. This means that if images are present in the markdown file, when jekyll builds the website, the images will be available to view freely. If the markdown contains many images, these may be sensitive and should not be viewable without a password. To get around this, I decided to embed the images in the page.

This means that the page may be larger but should load fairly quickly but most importantly, will not have the images present without a password.

When I write notes, I do so on a website called

Notion - The all-in-one workspace for your notes, tasks, wikis, and databases.

When I am finished writing, I can export to markdown with a directory with the same name and images inside that will be referenced. They will be referenced as follows:

![OpenKeyS%2085cb58b2e31b477499e994b152dd1a5e/Screenshot_at_2020-07-27_16-36-03.png](OpenKeyS%2085cb58b2e31b477499e994b152dd1a5e/Screenshot_at_2020-07-27_16-36-03.png)

Which is the markdown for an image with said image being in a different directory.

To insert an image in html, you would use syntax similar to:

<img src="picture.jpg" alt="Text if image doesn't render">

However, you can set the source of an image as a data uri for example:

<img src="data:<mime type>;base64,encoded data"

So for a png (default file type for screenshots on parrot os), the syntax might be:

<img alt="Image" src="data:image/png;base64,VGhpcyBpcyBub...3QgYW4gaW1hZ2U="

When the page loads, the image will load as usual but will make no additional requests so not only will the images not be stored but the page should load quickly!

The downside to this is that the files may become fairly large so I’d advise using a script to do this because copying this amount of text via a text editor makes the text editor fairly laggy.

My scripting language of choice is python and I’m sure there is probably a more efficient way to do it but it works so….

This function looks as follows:

def convertImages(filename):
    """
    assume directory of images has same name as name of markdown file
    each image has unique file name so regex not needed
    - convert images into base64 
    - replace references to images into data uri images 
    """
    
    markDown = open(filename,"r")
    content=markDown.read()
    markDown.close()

    
    #there must be a directory
    directory = filename[:-3]

    
    #images in markdown refernce a directory with escaped chars but python doesn't so convet
    namedDirectory = convertDirname(directory)

    #images in dir
    images = os.listdir(directory)

    #do for each image in directory
    for imageName in images:
        #convert image to b64 and replace in md file then delete said image
        with open(f"{directory}/{imageName}","rb") as image:
            encodedImage = b64encode(image.read())

            content=content.replace(f"![{namedDirectory}/{imageName}]({namedDirectory}/{imageName})",
                                    f'<img alt="Image" src="data:image/png;base64,{encodedImage.decode()}" />')
            os.remove(f"{directory}/{imageName}")                                    

    os.rmdir(directory)
    #used for testing output
    newContent=open(filename,"w")
    # newContent=open(filename,"w")
    newContent.write(content)
    newContent.close()

This searches the markdown and replaces any reference of the image with a base64 encoded version and then repeats this for every picture in the directory.

Of course this script could be modified to take multiple mime types (not just png) or different directories than pwd. However this gets the job done for what I need it to do since the screenshot tool only uses png.

To build the jekyll, this is as simple as calling the command! Either bundle exec jekyll build or jekyll build depending on what works for you (I am aware that jekyll build may not work for everyone and running this command suggests the former).

I again do this in python - by simply calling:

print("Jekyll Build - started")
os.system("bundle exec jekyll build --incremental")
print("Jekyll Build - finished")

Python waits for this command to finish before doing anything else so just print the start and then at end so a pause in the action of the script is not unexpected and is just the jekyll building.

I use incremental to save time by only building files that have changed. If you are intending to do the same, make sure to set regenerate: true in your index/post-listings file so that the unencrypted content isn’t leaked through jekyll having the content on the index page

Changing the template

Its fairly evident that you would not want the default staticrypt page to use as your password entry screen since it will most likely not match the theme you have made/are using. Fortunately, staticrypt has the ability to generate an encrypted string and the script that is bundled with the page can be used in your own template.

I will go through a rough idea of what this means so that you can make your own template.

Summary:

Generate a page with the components staticrypt needs and replace the default theme with your own

Ok, so staticrypt has a few components that are labeled in the template, these will be used by the script to retrieve the password a user enters and handle decrypting/showing content. I use all the components since I have no desire to change the script but if you do not want the components with the classes they do, feel free to change them - I understand the html more if I leave them.

You can still style the components to how you wish but if you want different class names, you will also have to modify the script.

The first of the scripts that are bundled with the template is crypto-js. This explains how the content is encrypted and decrypted, it uses the library to facilitate this. You can either leave this in your template to have the script embedded in each page you encrypt or you can have it elsewhere in your project repo and just reference it.

The script to decrypt looks as follows:

<script>

    /**
     * Decrypt a salted msg using a password.
     * Inspired by https://github.com/adonespitogo
     */
    var keySize = 256;
    var iterations = 1000;
    function decrypt (encryptedMsg, pass) {
        var salt = CryptoJS.enc.Hex.parse(encryptedMsg.substr(0, 32));
        var iv = CryptoJS.enc.Hex.parse(encryptedMsg.substr(32, 32))
        var encrypted = encryptedMsg.substring(64);

        var key = CryptoJS.PBKDF2(pass, salt, {
            keySize: keySize/32,
            iterations: iterations
        });

        var decrypted = CryptoJS.AES.decrypt(encrypted, key, {
            iv: iv,
            padding: CryptoJS.pad.Pkcs7,
            mode: CryptoJS.mode.CBC
        }).toString(CryptoJS.enc.Utf8);
        return decrypted;
    }

    document.getElementById('staticrypt-form').addEventListener('submit', function(e) {
        e.preventDefault();

        var passphrase = document.getElementById('staticrypt-password').value,
            encryptedMsg = '1f53b7fa...4a4a211fbd',
            encryptedHMAC = encryptedMsg.substring(0, 64),
            encryptedHTML = encryptedMsg.substring(64),
            decryptedHMAC = CryptoJS.HmacSHA256(encryptedHTML, CryptoJS.SHA256(passphrase).toString()).toString();

        if (decryptedHMAC !== encryptedHMAC) {
            alert('Bad passphrase!');
            return;
        }

        var plainHTML = decrypt(encryptedHTML, passphrase);

        document.write(plainHTML);
        document.close();
    });
</script>

So this script isn’t actually that complex and you could make improvements if you wish but I am going to leave it as default for the purpose of this article.
